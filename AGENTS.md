## React & Next.js Rules

- Functional components only. PascalCase file names (e.g., `MyComponent.tsx`).
- Server Components by default. Add `'use client'` only when interactivity/state/browser APIs are needed.
- Keep Client Components as small leaf nodes. Pass Server Components as `children` props when possible.
- Always provide stable, unique `key` props on list rendering.
- Always provide `useEffect` dependency arrays. Never omit.
- Use functional update form when new state depends on previous: `setCount(prev => prev + 1)`.
- Custom hooks: extract reusable stateful logic, name files starting with `use` (e.g., `useAuth.ts`).
- Use React Fragments (`<></>`) over wrapper divs when returning multiple elements.
- Server Actions via `'use server'` for mutations. Use `useFormState`/`useFormStatus` for form state.
- `useOptimistic` for immediate UI updates while Server Actions are pending.
- In Server Components, `await` data directly — never `useEffect` for data fetching.
- Dynamic routes use `[folder]` syntax, access via `props.params`.

## Coding Guidelines

These rules override generic preferences. Read before any implementation work. For full context, see the `karpathy-guidelines` skill.

### 1. Think Before Coding

Don't assume. Don't hide confusion. Surface tradeoffs. If something is unclear or multiple interpretations exist, stop and ask before implementing.

### 2. Simplicity First

Minimum code that solves the problem. Nothing speculative. No abstractions until there's real repetition. Push back on over-engineering.

### 3. Surgical Changes

Touch only what you must. Clean up only your own mess. Don't refactor unrelated code. Prefer targeted edits over rewriting.

### 4. Verify

Define success criteria before starting. Loop until verified. If acceptance criteria aren't met after implementation, the task isn't done.

### 5. Immutability

Always create new objects — never mutate existing ones. Return new copies with changes applied. Prefer `{...obj, field: newValue}` over `obj.field = newValue`. This prevents stale-reference bugs in React state and Convex mutations.

### 6. File Organization

Many small files over few large ones. 200–400 lines typical, 800 max. Organize by feature/domain, not by type. High cohesion, low coupling. A file should do one thing well.

### 7. Error Handling

Handle errors at every level. Provide user-friendly messages in UI code. Log detailed context server-side. Never silently swallow errors. Convex functions should throw `ConvexError` with meaningful messages.

### 8. Input Validation

Validate all user input at system boundaries. Use Convex `v.` validators on every function's `args`. Fail fast with clear messages. Never trust external data.

### Code Quality Checklist

Before submitting any code:
- Functions small (<50 lines), files focused (<800 lines)
- No deep nesting (>4 levels)
- Proper error handling, no hardcoded values
- Readable, well-named identifiers
- Immutability enforced — no direct mutation

## Security

**Before ANY commit, verify:**
- No hardcoded secrets (API keys, passwords, tokens)
- All user inputs validated (Convex validators on every function)
- XSS prevention (sanitized HTML, no `dangerouslySetInnerHTML` with untrusted input)
- CSRF protection enabled (Better Auth handles this)
- Authentication/authorization verified on protected routes
- Rate limiting on all public endpoints
- Error messages don't leak sensitive data (stack traces, internal IDs)

**Secret management:** NEVER hardcode secrets. Use environment variables or a secret manager. Validate required secrets at startup. Rotate any exposed secrets immediately.

**If a security issue is found:** STOP → use security-reviewer agent → fix CRITICAL issues → rotate exposed secrets → review codebase for similar issues.

## Design / UI

This project uses the **Impeccable** skill for design discipline. The four most common UI failures are: (1) no consistent theme across pages, (2) not creating reusable components, (3) not reusing existing components, (4) defaulting to AI-ish aesthetics. The rules below prevent all four.

### Mandatory: `/impeccable shape` before every new feature

**Before writing any UI code for a new page or feature**, run `/impeccable shape <feature>`. This produces a design brief through discovery: what's on the page, who the user is, what they're doing, what the edge cases are. No UI code should be written until the shape brief is confirmed. This prevents jumping straight to code and producing card grids by default.

If `PRODUCT.md` or `DESIGN.md` is missing or stale, run `/impeccable teach` (for PRODUCT.md) or `/impeccable document` (for DESIGN.md) **before** shaping.

### Component discipline

- **Check existing components before creating new ones.** Read DESIGN.md's Components section. If a Button, Card, Input, Badge, Dialog, or Table variant already exists, use it. Do not duplicate a component that differs only in padding, color, or border radius — extend the existing one with a variant prop.
- **Extract at 3+ uses.** If the same UI pattern appears three or more times with the same intent, run `/impeccable extract` to pull it into a shared component with proper variant/size props. Two usages are not a pattern. Three might be.
- **Use tokens, never hard-code.** Colors, spacing, radii, shadows, font sizes must reference CSS custom properties or Tailwind theme values defined in `globals.css` or `DESIGN.md`. No raw hex, no magic numbers.
- **Extend, don't duplicate.** When a page needs a slightly different Card or Button, add a variant to the existing component rather than creating `SpecialCard.tsx`. Variants stay inside the design system. Copies drift.
- **After building a new feature**, run `/impeccable polish` to catch hard-coded values, missing states (hover, focus, disabled, loading), and drift from the design system.

### Anti-patterns: forbidden by default

These are the marks of AI-generated UI. They are forbidden unless `PRODUCT.md` explicitly calls for them.

- **Purple/violet gradients** — the most recognizable AI tell. Use the project palette from DESIGN.md.
- **Glassmorphism / frosted-glass cards** — blur effects as decoration rather than real layering.
- **Nested cards** — cards inside cards. Use spacing, dividers, or typography for grouping.
- **Inter as the only font** — use the type pair from DESIGN.md. Pair a distinctive display font with a refined body font.
- **Icon tiles stacked above headings** — the universal AI feature-card template. Try side-by-side icon + heading, or no icon container at all.
- **Side-tab accent borders** — thick colored border on one side of a rounded card. Use a subtler accent or remove it.
- **Gradient text on headings** — decorative heading color. Use solid colors from the project palette.
- **Centered everything** — left-align body text. Center only heroes and short CTAs.
- **Identical card grids** — same-sized cards with icon + heading + text repeated endlessly. Break the pattern with visual variety.
- **Monotonous spacing** — the same gap value everywhere. Use tight groupings for related items, generous separations between sections.
- **Hero metric layouts** — big number, small label, stats row. Avoid unless the feature genuinely calls for it.
- **Overused fonts** — Inter, Roboto, Fraunces, Geist, Plus Jakarta Sans, Space Grotesk, Instrument Sans. Use the project's chosen pair from DESIGN.md instead.
- **Dark mode with glowing accents** — colored box-shadow glows on dark backgrounds. Use subtle, purposeful lighting instead.
- **Rounded rectangles with generic drop shadows** — commit to a stronger visual treatment or remove the shadow entirely.
- **Wrapping everything in cards** — not every piece of content needs a bordered container. Spacing and alignment create visual grouping without the overhead.

Run `npx impeccable detect app/` or `/impeccable audit` to catch these automatically.

### Workflow for every new page/feature

1. **Read `PRODUCT.md` and `DESIGN.md`** — load the design context before writing a single line of UI
2. **Run `/impeccable shape <feature>`** — mandatory discovery brief (who, what, edge cases, constraints)
3. **Check existing components** for reuse opportunity — consult DESIGN.md Components section
4. **Build**, referencing tokens from DESIGN.md — all colors, spacing, radii, shadows, type from the system
5. **Run `/impeccable polish`** for alignment pass — catch hard-coded values, missing states, design drift
6. **Run `/impeccable audit` or `npx impeccable detect app/`** before commit — anti-patterns block the commit automatically via Husky pre-commit hook
7. **If anti-patterns found**, fix before merging — the pre-commit hook will block the commit

### Key documents

- **PRODUCT.md** — strategic context: register (product), users, brand voice, anti-references, design principles
- **DESIGN.md** — visual system: colors, typography, elevation, components, do's and don'ts (Google Stitch format, six fixed sections)
- **Register:** product (design SERVES the task). All UI must respect design principles in PRODUCT.md.
- **Anti-references:** purple gradients, glassmorphism, nested cards, Inter font, hero banners, emoji copy, gratuitous shadows, decorative blurs

## Graphify — Code Structure

This project has an AST-only Graphify graph at `graphify-out/`. It tracks imports,
calls, and file relationships. No LLM enrichment — communities are unlabeled and
no wiki exists. The graph auto-rebuilds on commit (see `.husky/_/post-commit`).

If it seems stale, run `graphify update .` (AST-only, no API cost).

### What It's Good For

| Task | Tool |
|------|------|
| Trace dependencies between two symbols | `graphify path "auth.ts" "projects.ts"` |
| Blast radius before refactoring | `graphify path "<symbol>" "<downstream>"` |
| Visual exploration | Open `graphify-out/graph.html` |
| Manual rebuild | `graphify update .` |

### What It's Not Good For

- **Finding all references to a symbol** — use `rg`. The AST graph doesn't catch every usage.
- **Semantic questions** ("how does auth work?") — read the source files. The graph is structural, not conceptual.
